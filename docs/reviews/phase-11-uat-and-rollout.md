## Production rollout gate

1. Capture a fresh custom-format PostgreSQL backup and validate its restore on a
   disposable database.
2. Query `schema_migrations`; current production must have the known chain
   through `007_session_occurrences_and_pic_labels` before applying P11.
3. Apply migrations `008` through `015` using the restricted migration role.
4. Run `python scripts\phase8_automated_uat.py` against a disposable copy, then
   smoke-test the app using the restricted application role.
5. Generate `docs/reviews/phase-11-operational-issue-snapshot.json` from the
   final candidate database.
6. Resolve high-severity findings, or enter owner-approved decisions for
   supported legacy placeholder/exception cases in
   `docs/reviews/phase-11-owner-decision-template.json`.
7. Apply the owner template with `python
   scripts\phase11_operational_issue_snapshot.py --apply-decision-template`.
8. Run `$env:PHASE11_DB='english_class'; python
   scripts\phase11_operational_issue_snapshot.py --validate-decisions`; it
   must pass before rollout approval. Schedule conflicts must be corrected, not
   merely accepted while still present.
9. Keep the pre-rollout backup and legacy production available until the owner
   signs the operating-window review.

## Review answers

1. Expected: P11 migrations apply to production-shaped data without weakening
   source-of-truth constraints, and every detected issue is actionable.
2. Inspected: an eleven-migration disposable rehearsal, canonical row counts,
   restricted-role app smoke, backup/restore, and the issue breakdown above.
3. Traced: onboarding, roster, transfer, correction, two-test improvement,
   monthly export, and an incomplete employee correction fixture.
4. Automated evidence: Phase 6 security gate, Phase 7 workflow gate, Phase 8
   UAT, and Phase 9 cutover rehearsal.
5. Changed assumption: legacy records cannot all be treated as current active
   enrollments; ETL now retains history as completed and chooses the latest
   observed enrollment as active.
6. Remaining risk: the listed high-severity legacy records require owner data
   decisions before production rollout.
7. Approved legacy-attendance handling: an admin may approve a session-level
   exception only for a delivered session with missing applicable results. The
   exception removes that specific inbox finding but does not insert, update,
   or infer any `Present`/`Absent` attendance facts. When the owner confirms a
   shared legacy-source condition, the admin may apply it in bulk; every
   affected session still receives its own approval and audit event.
8. Approved legacy-placement handling: an admin may add `Unknown Entrance
   Level` only when no business placement exists. It is source-marked and
   audited, and never overwrites an observed entrance level.

## Rollback

Do not delete canonical rows or reverse individual migrations in production.
Take the app offline only if necessary, restore the validated pre-rollout dump
to the original database name, verify schema version and canonical counts, then
restart the restricted application role. Record the reason, actor, timestamps,
and restored backup checksum in the release log.
