# Project Engineering Rules

Status: **Mandatory for every implementation task**

These rules protect data correctness. A task is not complete because the code
runs or the UI looks correct. It is complete only when its output has been
reviewed, tested, and reconciled with the data contract.

## 1. Authority and source of truth

Use this precedence when documents or code disagree:

1. Confirmed business decisions in `DATA_DICTIONARY.md`.
2. Entity grain and invariants in `TARGET_ARCHITECTURE.md`.
3. Delivery sequence and acceptance criteria in `IMPLEMENTATION_PLAN.md`.
4. Existing production database behavior.
5. Legacy spreadsheet formulas and current code.

The workbook is a migration source, not the target model. A formula in Google
Sheets is not automatically a valid business rule.

When a requirement is ambiguous, stop and document the question. Do not encode
an assumption that changes entity grain, historical meaning, eligibility, or
reported metrics.

## 2. Grain before columns

Before creating or modifying a table, write one sentence:

> One row represents exactly ...

Every primary key, unique constraint, foreign key, and upsert key must match
that sentence. If the sentence is unclear, implementation must pause.

Do not use a composite key inherited from the spreadsheet without proving it
is unique for the target business process. In particular:

- `class_code + course_name` is not a course-run identity because a cohort can
  repeat a course.
- Employee name and PIC display name are never identifiers.
- Meeting duration and credited session units are separate concepts.
- Current state must not overwrite membership, evaluation, PIC, or BU/role
  history.

## 3. No silent data loss

Every source row must end in exactly one auditable outcome:

- loaded into a canonical target entity;
- retained in staging while awaiting another entity;
- recorded in `data_quality_issues` with a machine-readable reason; or
- intentionally ignored by an approved mapping rule.

Forbidden behaviors:

- dropping rows because a required key is missing without recording an issue;
- converting unmatched levels to null without preserving the original label;
- keeping the first duplicate without recording the competing rows;
- using `except: pass` or logging an error and continuing without a row outcome;
- claiming imported totals without reconciling them to source totals.

## 4. Preserve raw and historical data

- Raw imports are append-only and include import batch, source file checksum,
  sheet name, source row number, raw payload, and ingestion timestamp.
- Evaluation corrections create immutable versions.
- Transfers close an old enrollment and create a new one.
- BU/role changes create history; enrollment BU/role snapshots never change.
- Attendance corrections update the effective result and write an audit event.
- Migration scripts never delete source or legacy tables in the same release
  that first creates their replacement.

## 5. Input, snapshot, derived, and audit fields

Follow field classes in `DATA_DICTIONARY.md`:

- `input`: accepted only through validated commands/forms.
- `snapshot`: copied once at the defined business event.
- `derived`: calculated by SQL view/query; never manually editable.
- `audit`: generated by the system; never accepted from frontend input.

Do not add convenience copies such as employee name or PIC name to fact tables
unless a documented historical snapshot requires them.

## 6. Database rules

- PostgreSQL constraints enforce business invariants where practical.
- Use surrogate primary keys for canonical entities and business-key unique
  constraints for duplicate prevention.
- All foreign-key columns used in joins must have supporting indexes.
- Use `DATE` for date-only facts and `TIMESTAMPTZ` for real moments.
- Store timestamps in UTC; display in the configured business timezone.
- Use explicit check constraints or lookup tables for controlled statuses.
- Multi-table business operations use one transaction.
- Writes must be concurrency-safe; sequence generation and run numbering may
  not rely on `MAX(...) + 1` without locking or a database constraint.
- Application startup may check schema version but may not execute DDL.

## 7. Migration rules

- Never modify an applied migration.
- Before choosing a migration strategy, query `schema_migrations` on the target
  database and capture the result in the implementation review log.
- Back up the database before the first schema-changing command.
- Migrations must run with stop-on-error and one transaction when PostgreSQL
  supports it.
- Every migration includes a forward verification query and a rollback or
  restore procedure.
- Test both paths when relevant: fresh database and existing legacy database.
- Keep `DRAFT_MIGRATIONS.lock` until the complete migration chain passes all
  required gates.

## 8. ETL and data-quality rules

- ETL is idempotent for the same source checksum.
- Re-running a completed batch must not create duplicate canonical records.
- Transformation functions are deterministic and independently testable.
- Source values are normalized only through documented mappings.
- Quality issue types are stable codes, not free-form error messages.
- Resolved issues retain original details, resolution action, actor, and time.
- Schedule-dependent KPIs exclude unresolved structural attendance anomalies.

Required reconciliation equation per sheet:

```text
source data rows
= canonical rows represented
+ unresolved issue rows
+ approved ignored rows
```

The terms must be defined at the same source-row grain.

## 9. Business-rule implementation

- Attendance eligibility uses applicable, non-cancelled session units.
- Sessions before an enrollment's `start_session_number` are not applicable,
  not absent.
- An admin override requires actor, reason, previous result, and timestamp.
- A make-up links one later Present event to one original Absent row. The
  original remains unchanged, the linked credit does not add a denominator
  unit, and the change is retained in audit history.
- Pass and next-course eligibility come from the teacher evaluation.
- Current level means the latest valid evaluation; highest level is a separate
  metric. Never substitute one for the other.
- Coverage KPI remains unavailable until a complete HR denominator exists.

## 10. Security and privacy

- Never commit database passwords or `.streamlit/secrets.toml`.
- The app uses a restricted database role, never the PostgreSQL superuser.
- Parameterize every query containing external or user-provided values.
- Do not expose password hashes, connection strings, raw SQL consoles, or
  internal audit payloads to unauthorized users.
- Authorization is enforced in the service layer as well as the UI.
- Sensitive actions include actor identity in the audit log.

## 11. Code structure

- Keep database access and business commands outside Streamlit page code.
- UI modules collect input and render results; they do not own SQL strings.
- Repository functions handle persistence, service functions handle business
  transactions, and views/queries handle reporting.
- Avoid generic helpers that hide transaction boundaries or entity grain.
- Use clear business names: `course_run`, not `offering` when the entity is a
  cohort taking a course one time.
- Comments explain non-obvious business constraints, not obvious syntax.

## 12. Required build-review loop

Every task follows this loop. Skipping a step is not allowed.

1. **Read:** Re-read the relevant data dictionary and architecture sections.
2. **State grain:** Write the row grain and invariants affected by the task.
3. **Profile:** Inspect real source examples and known anomalies.
4. **Implement small:** Make the smallest coherent change.
5. **Self-review:** Review the full diff for wrong assumptions, destructive
   behavior, missing constraints, duplicated data, and security leaks.
6. **Test:** Run unit, database, migration, and/or UI tests appropriate to risk.
7. **Reconcile:** Compare output counts and representative records with source
   data and expected business behavior.
8. **Adversarial check:** Test at least one invalid input and one historical or
   edge-case scenario.
9. **Record evidence:** Complete `DEVELOPER_REVIEW_CHECKLIST.md` in the PR/task
   notes with commands and important output.
10. **Review again:** Re-open generated SQL/views/UI output after tests. Do not
    assume a passing test means the output is semantically correct.

## 13. Definition of done

A task may be marked complete only when all applicable conditions are true:

- acceptance criteria are demonstrated;
- output was manually reviewed after generation;
- automated tests pass;
- source-to-target reconciliation passes;
- no source row is silently lost;
- known anomalies are preserved or surfaced;
- migrations were tested on disposable databases;
- documentation changed with the behavior;
- security and permissions were reviewed;
- residual risks and deferred cases are explicitly listed.

If evidence is missing, the task status is `in progress`, not `complete`.
